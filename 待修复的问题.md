### 🔴 CRITICAL-01：英文模式下 `text.usetex=True` 无条件开启，LaTeX 未安装时必崩

**位置：** `_core/style.py` → `setup_style()`

**代码：**
```python
if lang == "en":
    plt.rcParams["text.usetex"] = True   # ← 无条件开启
else:
    plt.rcParams["text.usetex"] = False
```

**问题：** 调用 `setup_style(lang="en")` 不报错，但 `fig.savefig()` 或 `plt.show()` 时 matplotlib 会调用 LaTeX 进程，如果系统没装 LaTeX（大多数用户），会抛出：
```
FileNotFoundError: [WinError 2] The system cannot find the file specified
RuntimeError: Failed to process string with tex
```
错误发生在渲染时而不是设置时，用户根本不知道是 `setup_style` 出的问题。

**修复：**
```python
if lang == "en":
    # 检测系统是否有 LaTeX，没有则静默降级
    import shutil
    if shutil.which("latex") or shutil.which("xelatex"):
        plt.rcParams["text.usetex"] = True
    else:
        plt.rcParams["text.usetex"] = False
else:
    plt.rcParams["text.usetex"] = False
```

### 🔴 CRITICAL-02：`plot_heatmap` 的 vmin/vmax 与 `**kwargs` 存在参数冲突

**位置：** `_plots/advanced.py` → `plot_heatmap()`

**代码：**
```python
im = ax.imshow(data, cmap=cmap, aspect="auto", vmin=None, vmax=None, **kwargs)
```

**问题：** 如果用户传入 `vmin=0` 或 `vmax=1`（热力图常见需求）：
```python
sp.plot_heatmap(data, vmin=0, vmax=1)   # ← 这会 TypeError
```
`vmin` 和 `vmax` 同时存在于显式参数和 `**kwargs` 中，Python 会报：
```
TypeError: imshow() got multiple values for keyword argument 'vmin'
```

**修复：** 把 `vmin`/`vmax` 提到函数签名里：
```python
def plot_heatmap(
    data, ...,
    vmin: Optional[float] = None,
    vmax: Optional[float] = None,
    **kwargs,
) -> PlotResult:
    ...
    im = ax.imshow(data, cmap=cmap, aspect="auto", vmin=vmin, vmax=vmax, **kwargs)
```

### 🔴 CRITICAL-03：调用任意绘图函数都会静默重置用户设置的 `lang`

**位置：** `_core/utils.py` → `apply_resolved_style()` → `setup_style()`

**现象：**
```python
sp.setup_style("ieee", "earth", lang="en")   # 用户设置英文 IEEE 样式
fig, ax = sp.plot(x, y)                      # 内部调用 apply_resolved_style(None, None)
                                              # → setup_style("nature", "pastel")
                                              # lang 被重置为默认 "zh"，字体变回宋体
```

**根因：** `apply_resolved_style()` 调用 `setup_style()` 时没有传 `lang`：
```python
def apply_resolved_style(venue, palette):
    ...
    setup_style(effective_venue, effective_palette)  # lang 永远用默认值 "zh"
```

**影响：** 用户无法通过 `setup_style(lang="en")` 持久化语言设置，必须用 `StyleContext`，而文档中没有强调这一限制。

**修复：** 在全局存储当前 lang 状态，或让各绘图函数接受 `lang` 参数：
```python
# 方案A：全局记录 lang
_current_lang: Optional[str] = None

def setup_style(venue="nature", palette="pastel", lang="zh"):
    global _current_lang
    _current_lang = lang
    ...

def apply_resolved_style(venue, palette):
    ...
    setup_style(effective_venue, effective_palette, lang=_current_lang or "zh")
```

### 🔴 CRITICAL-04：`plot_horizontal_bar(sort=True)` 排序方向相反

**位置：** `_plots/distribution.py` → `plot_horizontal_bar()`

**代码：**
```python
if sort:
    sorted_indices = np.argsort(values)[::-1]   # 降序
    categories = [categories[i] for i in sorted_indices]
    values = np.array(values)[sorted_indices]
```

**问题：** `barh` 的 y=0 在**底部**，y=N-1 在**顶部**。降序排列后，最大值对应 y=0，即出现在图表**最底部**。用户期望"排序"意味着最重要/最大的条目显示在最顶部，但实际结果相反。

验证：
```python
sp.plot_horizontal_bar(["A","B","C"], [30, 10, 20], sort=True)
# 显示（从上到下）: C(20), A(30), B(10) ← 不是降序！
# 实际最大值 A 在最底部
```

**修复：** 升序排列让最大值在顶部：
```python
if sort:
    sorted_indices = np.argsort(values)   # 升序（barh 中最大值在顶部）
```

### 🔴 CRITICAL-05：`PlotResult.__exit__` 自动关闭图形，上下文外无法保存

**位置：** `_core/result.py` → `PlotResult.__exit__()`

**代码：**
```python
def __exit__(self, ...):
    self.close()   # ← 关闭 Figure
    return False
```

**问题：**
```python
with sp.plot(x, y) as result:
    result.ax.set_xlabel("X")

result.save("fig")   # BUG：figure 已被 close()，这里会保存空白图或报错
```

这是"聪明反被聪明误"的设计，使用上下文管理器的正常语义是"进入时获取资源，退出时释放"，但绘图场景更常见的用法是"在 with 块里设置属性，退出后保存"。自动关闭会让这个常见用法变成陷阱。

**修复：** 移除 `__exit__` 中的自动 close，或明确文档告知此风险，并提供 `auto_close=False` 选项：
```python
def __init__(self, fig, ax, metadata=None, auto_close=False):
    self._auto_close = auto_close
    ...

def __exit__(self, ...):
    if self._auto_close:
        self.close()
    return False
```

### 🟠 HIGH-01：`lang` 参数不对称——plotting 函数接受 venue/palette 但没有 lang

**现象：**
```python
# venue 和 palette 可以在每次调用时指定
sp.plot(x, y, venue="ieee", palette="earth")  # ✓ 有效

# 但 lang 没有这个能力——只能用全局 setup_style
sp.plot(x, y, lang="en")   # ✗ TypeError，参数不存在
```

**影响：** 在同一个脚本里需要中英混排（如中文正文图 + 英文投稿图）时，用户必须反复调用 `setup_style(lang=...)` 或嵌套 `StyleContext`，API 的一致性被破坏。

**修复：** 给所有绘图函数添加 `lang: Optional[str] = None` 参数，在 `apply_resolved_style()` 中统一处理。

---

### 🟠 HIGH-02：`create_subplots`/`paper_subplots`/`create_gridspec` 内部 setup_style 静默重置配色

**位置：** `_core/layout.py`

**代码：**
```python
def create_subplots(nrows, ncols, venue="nature", ...):
    from sciplot._core.style import setup_style
    setup_style(venue)   # ← 只传 venue，palette/lang 重置为默认值
    ...
```

**问题：**
```python
sp.setup_style("ieee", "earth", lang="en")   # 用户设定

fig, axes = sp.paper_subplots(2, 2, venue="ieee")
# 内部调用了 setup_style("ieee")
# → palette 重置为 "pastel"，lang 重置为 "zh"
# 用户的 "earth" 配色和 "en" 语言设置消失了
```

**修复：** `create_subplots`/`paper_subplots`/`create_gridspec` 应该接受并传递 `palette` 和 `lang` 参数，或者只调用 `apply_palette(palette)` 而不调用完整的 `setup_style()`。

### 🟠 HIGH-03：`plot_line` 与 `plot_multi_line` 的图例行为不一致

**现象：**
```python
# plot_line：有 label 才显示图例
sp.plot(x, y)                  # 无图例（正确）
sp.plot(x, y, label="信号")    # 有图例（正确）

# plot_multi_line：始终显示图例（即使是自动生成的 "Series 1"）
sp.plot_multi(x, [y1, y2])     # 显示 "Series 1, Series 2"（用户没要求）
```

单系列和多系列函数对"什么时候显示图例"的规则不一样，API 行为不统一。

**修复：** `plot_multi_line()` 应在 `labels` 为 `None`（即自动生成）时不显示图例，或添加 `show_legend: bool = True` 参数让用户控制：
```python
def plot_multi_line(..., show_legend: bool = True, ...):
    ...
    if show_legend:
        ax.legend()
```

### 🟠 HIGH-05：`PlotResult` 的方法对单/多子图的支持不对称

**现象：**
```python
result = sp.paper_subplots(1, 3, venue="thesis")

# 这些方法在多子图下有效（遍历所有轴）
result.xlabel("时间 (s)")   # ✓ 对所有子图设置
```

2. 或统一改为对多子图全部报错（要求用户 `result.axes[i].method()`），并在文档中明确说明
3. 保持现状但在文档中用表格明确列出哪些方法支持多子图

### 🟠 HIGH-08：`create_twinx()` 文档示例顺序错误，教用户坏习惯

**位置：** `_core/layout.py` → `create_twinx()` docstring

```python
def create_twinx(ax: Axes) -> Axes:
    """
    示例:
        >>> fig, ax1 = sp.new_figure("ieee")
        >>> sp.setup_style("ieee", "pastel-2")   # ← 创建图形之后才设置样式！
        >>> ax1.plot(x, temp, ...)
    """
```

正确顺序应该是先 `setup_style`，再 `new_figure`。样式在图形创建时就被应用到 rcParams，创建图形后再调用 `setup_style` 会触发 `plt.rcdefaults()`，这不会追溯影响已创建的图形，但会让状态混乱。

**修复：** 调整文档示例，先 style 后 figure：
```python
>>> sp.setup_style("ieee", "pastel-2")
>>> fig, ax1 = sp.new_figure("ieee")
>>> ax2 = sp.create_twinx(ax1)
```

### 🟡 MEDIUM-01：`annotate_significance()` 的 `h` 参数单位描述错误

**位置：** `_plots/distribution.py` → `annotate_significance()` docstring

**文档：**
```
h  : 括号高度（axes 单位），默认 0.02
```

**实际：** `h` 是**数据坐标**单位，不是 axes 单位（axes 单位是相对于坐标轴范围的 0-1 值）。

**影响：** 对于 y 轴范围是 [0, 100] 的图，`h=0.02` 意味着括号高度只有数据范围的 0.02，几乎不可见。用户会以为是 axes 单位（即轴高度的 2%，应该清晰可见），结果困惑为什么括号看不到。

**修复：** 文档改为"数据坐标单位"，并在示例中加说明：
```
h  : 括号高度（数据坐标，与 y 轴数据单位相同）。
     默认 0.02，适合 y 轴范围为 [0, 1] 的场景。
     y 轴范围较大时（如 [0, 100]），建议设为 y_range * 0.03 左右。
```

### 🟡 MEDIUM-02：`plot_confidence()` 的 `label_std` 默认值不跟随 `n_std`

**位置：** `_plots/advanced.py` → `plot_confidence()`

**问题：**
```python
# 用户想画 95% 置信区间
sp.plot_confidence(x, mean, se, n_std=1.96)
# 图例显示："±1σ"  ← 错误！应该是 "95% CI"
```

`n_std=1.96` 明确表示 95% CI，但图例还是默认的 `"±1σ"`，图例与实际含义不符，会导致论文图表标注错误。

**修复：** 让 `label_std` 能感知 `n_std`：
```python
def plot_confidence(
    ...,
    n_std: float = 1.0,
    label_std: Optional[str] = None,   # 改为 Optional，None 时自动生成
    ...
):
    if label_std is None:
        if abs(n_std - 1.96) < 0.01:
            label_std = "95% CI"
        elif abs(n_std - 2.576) < 0.01:
            label_std = "99% CI"
        elif abs(n_std - 1.0) < 0.01:
            label_std = "±1σ"
        else:
            label_std = f"±{n_std:.2g}σ"
```

## 第四章：静默错误与边界条件陷阱

### 🟡 MEDIUM-08：`PlotResult.save()` 始终调用 `tight_layout()`，无法被覆盖

**位置：** `_core/result.py` → `PlotResult.save()`

```python
def save(self, name, dpi=1200, formats=("pdf","png"), ...):
    self._fig.tight_layout()   # ← 无条件调用
    return _save(...)
```

**问题：** 用户精心调用了 `fig.subplots_adjust(hspace=0.5, bottom=0.15)` 来手动控制间距，然后调用 `result.save()` 时 `tight_layout()` 会覆盖用户的设置，导致出图与预期不符。

这是一个静默错误——图能保存，但布局不是用户想要的。

**修复：** 添加 `tight: bool = True` 参数：
```python
def save(self, name, ..., tight: bool = True, **kwargs):
    if tight:
        self._fig.tight_layout()
    return _save(...)
```

---

### 🟡 MEDIUM-09：`plot_errorbar()` 对 x/y/yerr 长度不做任何验证

**位置：** `_plots/advanced.py` → `plot_errorbar()`

所有其他函数（`plot_line`、`plot_scatter` 等）都有明确的长度校验，唯独 `plot_errorbar` 没有。传入长度不匹配的 x/y/yerr 会得到 matplotlib 底层的报错，错误信息晦涩难懂。

### 🟡 MEDIUM-11：`_is_dark_color()` 不验证输入，传入 named color 会崩

**位置：** `_plots/distribution.py`

```python

    h = hex_color.lstrip("#")
    r, g, b = int(h[0:2], 16), int(h[2:4], 16), int(h[4:6], 16)   # ← 会崩
```

当 `color` 是 matplotlib 命名颜色（如 `"blue"`、`"tab:red"`）而不是 hex 格式时，`h[0:2]` 会是 `"bl"`，`int("bl", 16)` 报 `ValueError`。

**修复：**
```python
def _is_dark_color(color: str) -> bool:
    try:
        import matplotlib.colors as mcolors
        rgb = mcolors.to_rgb(color)   # 支持所有 matplotlib 颜色格式
        r, g, b = rgb
        brightness = (r * 299 + g * 587 + b * 114) / 1000
        return brightness < 0.502
    except (ValueError, AttributeError):
        return False   # 解析失败时保守地返回 False（白色文字）
```

### 🟡 MEDIUM-16：`PlotChain.area()` 将 `alpha` 同时应用于线和填充区域

### 🟢 LOW-03：`validate_array_like()` 将 ndarray 转换为 list 会丢失 dtype 信息

**位置：** `_core/utils.py`

```python
def validate_array_like(data, name, ...):
    ...
    if isinstance(data, np.ndarray):
        result = data.tolist()   # ← dtype 丢失，float32 变 Python float，int64 变 int
    ...
    return result   # 返回 list，不是 ndarray
```

调用方接收到的是 Python 原生 list，而不是 numpy array。在 `plot_bar()` 里：
```python
values = validate_array_like(values, "values")   # 现在是 list
ax.bar(categories, values, ...)   # matplotlib 能接受，但丢失了 dtype
```

虽然在这里无害，但 `validate_array_like` 的语义应该是"验证"而不是"转换类型"，它悄悄改变了数据的类型，这违反了最小惊讶原则。

**修复：** 返回原始数据（只做验证不做类型转换），或明确改名为 `coerce_to_list()`：
```python
def validate_array_like(data, name, ...):
    ...
    # 只验证，不转换
    if hasattr(data, '__len__'):
        length = len(data)
    else:
        try:
            data = list(data)
            length = len(data)
        except TypeError:
            raise TypeError(...)
    
    if not allow_empty and length == 0:
        raise ValueError(...)
    
    return data   # 返回原始数据
```

### 🟢 LOW-04：函数内部的延迟 import 是规避循环导入的代码坏味道

**位置：** 多处

```python
# layout.py
def create_subplots(...):
    from sciplot._core.style import setup_style   # 函数内 import

# utils.py
def apply_resolved_style(...):
    from sciplot._core.style import setup_style
    from sciplot._core.palette import DEFAULT_PALETTE
```

函数内 import 是为了避免循环导入（`style` 引用 `palette`，`palette` 引用 `style` 等）。这种方式虽然有效，但：
1. 每次调用函数都要执行 import 语句（即使 Python 缓存了模块，仍有 sys.modules 字典查找开销）
2. 模块依赖关系变得隐藏，`import` 语句本应是模块依赖的显式声明
3. 增加了认知负担，阅读代码时看不出模块间的依赖

**修复：** 整理模块依赖层级，彻底消除循环导入：
```
_core/palette.py     → 不依赖其他 _core 模块（独立）
_core/style.py       → 只依赖 palette.py
_core/layout.py      → 依赖 style.py
_core/utils.py       → 依赖 style.py、palette.py、context.py
```

### 🟢 LOW-06：`plot_heatmap()` 的 `aspect="auto"` 硬编码，不适合方阵热力图

**位置：** `_plots/advanced.py`

```python
def plot_heatmap(..., aspect: str = "auto", ...):
```
